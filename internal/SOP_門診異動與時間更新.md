# SOP：門診異動與門診時間更新

> **用途**：立欣診所官網「門診時間相關變更」的標準作業流程。涵蓋兩種情境，動手前**先判別屬於哪一種**，再照對應清單執行。
> **權威順位**：本 SOP 是執行手冊；事實與合規規則仍以 `00_專案總覽索引.md`（§4 合規、§8 文件維護）為準，判準以 `02_優化決策手冊.md`（R4 dateModified）為準。兩者衝突時以 00／02 為準並回修本檔。
> **重要限制**：`validate_site.py` **不檢查門診時間一致性**（只驗 HTML/JSON-LD 結構、合規禁語）。因此情境 B 的全站同步**全靠本清單防漏**，漏一處就會前後矛盾（對家長誤導、對 SEO/AEO 是硬傷）。

---

## 〇、先判別情境（決策樹）

```
門診時間有變 →
├─ 只是「某一天」臨時調整（醫師進修/國定假日/颱風/單日停診）
│   且下週起恢復原狀？
│        → 情境 A：臨時單日異動（公告圖 + EXCEPTIONS + 自動過期）
│
└─ 是「常態」改變固定週班表（新增/取消某時段、改起訖時間、改休診日）？
         → 情境 B：常態門診時間變更（改 SCHEDULE + 全站同步）
```

兩者可能同時發生（例：宣布下個月起週日改休診，本週日先停一次）——先照 B 改常態，再視需要用 A 補當週公告。

---

## 一、情境 A：臨時單日異動

**成果樣貌**：首頁「最新門診異動」區塊(`#clinic-notice`)換上新公告圖、HERO 徽章當日顯示異動後時段、到期自動下架。

### 步驟

1. **公告圖**（正方形）放入 `images/notice/`，**檔名帶日期**（如 `clinic-notice-20260819.webp`＋`.jpg`），避免同名檔被 CDN 快取舊圖。
   - 若院長只給對話貼圖、無原始檔：用 `internal/tools/make_infographic.py`（或 Playwright 截圖＋PIL/sharp）重製 1000² webp＋jpg 後備。
2. **改 `index.html` 的 `#clinic-notice` 區塊**（搜 `id="clinic-notice"` 定位），共五處：
   - 圖片路徑 **兩處**（區塊本體 `#clinic-notice-zoom` 內 `<picture>` + lightbox `#notice-lightbox` 內 `<picture>`，webp source 與 jpg img 都要改）
   - `alt`（**公告全文逐字寫入**，爬蟲/螢幕閱讀器讀不到圖內文字）
   - 適用日期（`<time datetime>` 與可見「適用日期」文字）
   - 圖下可見說明文字（AEO：把圖內關鍵資訊重述一遍）
   - `data-expires="YYYY-MM-DD"`（適用末日，翌日起 UTC+8 自動隱藏整個區塊）
3. **若異動影響「當日開診時段」**（提早截止、當日某段停診、整日休診）→ 同步改 HERO 徽章的 `EXCEPTIONS` 表（`index.html` 內嵌 script，搜 `var EXCEPTIONS`）：
   - 鍵＝UTC+8 日期 `YYYY-MM-DD`，值＝**當日完整時段表**（整筆取代該日 `SCHEDULE`）
   - 格式：`[[開始分鐘,結束分鐘,'名稱'], …]`；分鐘＝時×60＋分（08:00=480、11:00=660、11:30=690、12:00=720、14:30=870、18:00=1080、18:30=1110、21:00=1260、21:30=1290）
   - **整日休診 = 空陣列 `[]`**（徽章自動顯示「目前休診，下個有診日…」）
   - 過期條目不影響運作，下次更新公告時順手清掉舊的即可
4. **收尾**（見 §三 共同尾段）：dateModified 跳 + sitemap 首頁 lastmod + 驗證 + Chromium 實測 + 部署。**臨時公告不進 llms**（時效資訊，過期即誤導 AI 引用）。

### 預設交付模式：即刻公告 + 立即部署（院長 2026-07-23 常設指示）

當**院長在對話直接貼上符合情境 A 的公告圖**時，預設一條龍走完、**不必逐次徵詢文案核可或部署確認**：

- **即刻公告**：公告區塊**不設起始日**，部署後即時顯示（現行實作即如此，只靠 `data-expires` 於「適用末日翌日」UTC+8 自動下架）。即使異動是未來日期，也是貼上當下就開始預告。
- **立即部署**：完成本節編修並通過 §三 驗證（`validate_site.py --stage deploy` ERROR 清零；動到徽章則 Chromium 實測四態）後，**直接快進合併進 `main` 並 `git push origin main`**（Netlify 由 `main` 建置上線），不再另開一輪問「文案可以嗎／要不要部署」。
- **文案處理**：公告可見文字（alt／圖下說明）就是**逐字轉述院長自製海報**，視同已核可；部署後於回覆中**附上實際上線之可見文案**供院長事後檢視。
- **仍不可放行的紅線**（照舊把關，發現即停下問院長）：合規禁語（`00 §4`，不得夾帶療效宣稱／自稱超級詞）；驗證器 ERROR；徽章實測未過。
- **邊界**：本預設**僅限情境 A 的對話貼圖公告**。情境 B（常態門診時間變更）為多頁散改、屬可見醫療/營運資訊，**維持逐字核可**、不套用即刻部署。

> ⚠️ 本則覆寫 `CLAUDE.md`／`00` 對「可見文字逐字核可」在**情境 A 貼圖公告**這一狹義情形的前置卡關；合規紅線與驗證關卡不受影響。

## 二、情境 B：常態門診時間變更

**成果樣貌**：全站每一處門診時間敘述、結構化資料、徽章班表都一致改成新時間。

> ⚠️ **這是「一改要同步十幾處」的高風險工作。** 先跑探查指令抓出所有出現點，再逐一對照下表，最後回頭複跑探查指令確認沒有殘留舊時間。

### 探查指令（先跑，掌握全貌）

```bash
# 抓所有時段數字出現處（排除 archive 歷史檔）
grep -rnE '18:30|21:30|08:00|11:30|14:30|12:00' --include='*.html' --include='*.txt' . \
  | grep -v 'internal/archive'
# 抓固定敘述句（改時間時這類句子最容易漏）
grep -rn '平日夜診至21:30\|夜診至 21:30\|週六週日皆有門診' --include='*.html' --include='*.txt' . \
  | grep -v 'internal/archive'
```

### 必改清單（`index.html`）

| 位置 | 定位方式 | 內容 |
|---|---|---|
| **JSON-LD `openingHoursSpecification`** | 搜 `openingHoursSpecification`（約 165–212 行） | 機器可讀的門診時間權威來源。每段 `opens`/`closes`/`dayOfWeek`。**AI 與 Google 讀這裡，最優先改。** |
| **PropertyValue 描述** | 搜 `平日夜間門診`／`週六兒科時段`（約 502–505 行） | 「週一至週五 18:30–21:30」等文字值 |
| **首頁門診時間表格** | 搜 `id="schedule"`（約 2459–2544 行） | 可見表格，逐格時間 |
| **HERO 徽章 `SCHEDULE` 表** | 搜 `var SCHEDULE`（約 5269 行） | 開診狀態判斷來源。鍵 0=日、1–6=一～六；格式同 §一.3。改完 Chromium 實測。 |
| **FAQ／正文寫死時段** | 搜 `18:30`／`21:30`（約 1814、2588、3126、3905–4006 行等多處） | 各 FAQ 答案、正文段落 |
| **meta / og / twitter description** | 搜 `平日夜診至21:30`（14、22、36–45 行等） | 介紹層敘述 |

### 必改清單（其他頁面）

| 檔案 | 位置 | 內容 |
|---|---|---|
| `visit-guide.html` | JSON-LD description（約 784）、FAQ 答案（約 831）、meta desc（14/21/28） | 門診時間表與敘述 |
| `services/weekend-pediatrics.html` | JSON-LD `openingHoursSpecification`（約 78+）、description（約 62）、meta desc（7/16） | 假日/夜間時段（此頁專講假日夜診，務必改） |
| **其他 services/ 與 health/ 頁** | 探查指令列出的 meta desc | 多頁 desc 含「平日夜診至21:30」固定句 |
| `llms.txt`／`llms-full.txt` | 搜門診時間相關條目 | 提供給 LLM 的摘要，須同步 |

### 站外（提醒院長，session 無法直接改）

- **Google 商家檔案**：營業時間（AI/地圖直接讀，影響最大）。
- **MainPi 佈告欄／AI 提示詞**：`internal/mainpi/` 有稿；門診時間變更時同步更新後請院長貼入後台。

### 收尾

見 §三 共同尾段。常態變更是**可見醫療/營運資訊**，dateModified 跳、需院長逐字核可、涉及多頁 → 部署後**補送 IndexNow**（`internal/tools/submit_indexnow.py`）。

## 三、共同尾段（兩情境都要做）

1. **dateModified／sitemap（R4）**：門診時間屬「家長讀到的營運資訊」→ 變更頁 `dateModified` 跳（完整 ISO 8601＋08:00）、`sitemap.xml` 對應 `lastmod` 同步。`lastReviewed` 不動（非醫師重審醫療內容）。判準見 `02` R4。
2. **合規**：時間敘述不得夾帶自稱超級詞、療效宣稱；紅線見 `00 §4`。
3. **驗證器**：`python3 internal/tools/validate_site.py --stage deploy` → **ERROR 清零才 push**。
4. **實機驗證**：改動 HERO 徽章（`SCHEDULE`/`EXCEPTIONS`）時，用 Chromium 模擬時間實測（腳本範例：`Date.now` 覆寫成目標時刻，讀 `#clinic-status-text`），確認「開診中/即將/休診/下個時段」四態正確、跨日正確、特例生效。
5. **院長核可**：可見文字（公告文案、表格時間、FAQ 敘述、徽章訊息）逐字經院長核可才 push（`00` 鐵律）。
6. **文件維護（§8）**：改 `00`/`01` 前先 `cp` 備份至 `archive/`；本批於 `00` 文末 append 附錄、檔頭更新版本行、`01` 同步。
7. **部署後**：內容實質變更批（尤其情境 B 多頁）→ 補送 IndexNow；情境 A 單頁公告可省。

## 四、常見錯誤（照做即可避開）

| 錯誤 | 後果 | 防法 |
|---|---|---|
| 情境 B 只改了首頁表格，漏改 JSON-LD | AI/Google 引用到舊時間 | 先跑探查指令、改完回頭複跑確認零殘留 |
| 公告圖用同檔名覆蓋 | CDN 快取舊圖、家長看到過期資訊 | 檔名一律帶日期 |
| 臨時異動只改公告圖、沒改 EXCEPTIONS | 徽章與公告矛盾（徽章仍顯示原時段） | §一.3：影響當日時段就加 EXCEPTIONS |
| 忘記設 data-expires | 過期公告一直掛首頁誤導家長 | §一.2：一律設 `data-expires` |
| 臨時公告寫進 llms | 過期後 AI 仍引用 | 臨時公告不進 llms |
| 改了時間沒跳 dateModified／sitemap | SEO 訊號過期 | §三.1（R4） |
